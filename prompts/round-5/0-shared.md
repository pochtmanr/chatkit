## 1. The mental model

Round 5 splits the widget into two cleanly separated surfaces and
introduces a real user identity for the customer surface.

```
                      ┌────────────────────┐
                      │  Host app backend  │   has sk_live_…
                      └─────────┬──────────┘
                                │  POST /api/v1/widget-tokens
                                │  Bearer sk_live_…
                                ▼
                      ┌────────────────────┐
                      │  Holylabs platform │   signs HS256 JWT
                      └─────────┬──────────┘
                                │  returns { token, expires_at }
                                ▼
   ┌──────────────┐     iframe URL: /embed/customer
   │ Host browser │ ─── + bearer JWT for every API call
   └──────────────┘
          │
          │  postMessage (handshake nonce, then narrow command set)
          ▼
   ┌──────────────────────────────────────────────────────┐
   │  /embed/customer/*  +  /api/embed/customer/*         │
   │  ── reads/writes scoped to JWT.sub                   │
   │  ── never lists tenant-wide conversations            │
   └──────────────────────────────────────────────────────┘

   ┌──────────────────────────────────────────────────────┐
   │  /embed/inbox/*   +   /api/embed/agent/*  (501 stub) │
   │  ── round 6 surface; round 5 only draws the border   │
   └──────────────────────────────────────────────────────┘
```

Three distinct credential classes:

- **`pk_live_…` / `pk_test_…`** — publishable, browser-safe.
  Identifies the inbox. Already exists; round 5 narrows its
  authority: it can only be used to bootstrap the widget and to
  attach an inbox to a widget JWT. It cannot list conversations.
- **`sk_live_…`** *(new)* — server secret. Held by the host
  backend. Used solely to call `/api/v1/widget-tokens` and (in
  round 6) other server APIs. Hashed at rest. Shown once at
  creation. Dual-key rotation.
- **Widget JWT** *(new)* — short-lived HS256 token issued by us,
  identifying one end user inside one inbox. Verified on every
  `/api/embed/customer/*` call.

## 2. Key formats and storage

### 2.1 `pk_…` (unchanged)

Continues to live in `inboxes.api_key` (text), generated by the
existing inbox creation flow. Visible in the dashboard.

### 2.2 `sk_live_…` *(round 5)*

```
sk_live_<32-char-hex>          example: sk_live_d3b0c44298fc1c149afb...
```

Storage on `inboxes`:

- `server_secret_hash` — `text`. SHA-256 of the raw key, base64.
- `server_secret_previous_hash` — `text` nullable. Predecessor
  for rotation grace.
- `server_secret_rotated_at` — `timestamptz` nullable. Set when
  `previous_hash` was last populated. Past `+ 24h`, the previous
  key stops verifying (a cron in round 6 will sweep; round 5
  enforces in the verify helper).

Constraints:

- `server_secret_hash <> server_secret_previous_hash` when both
  are non-null (mirrors the round-4 `inbox_webhook_secret_distinct`
  check). See `supabase/migrations/0024_inbox_webhook_signing.sql`
  for the exact pattern.

A `lookupServerSecret(rawKey)` helper resolves an `sk_live_…` to
an `{ inboxId, businessId }` by hashing and matching against
either column.

### 2.3 Widget JWT *(round 5)*

```jsonc
header:  { "alg": "HS256", "typ": "JWT", "kid": "<inbox.id>" }
payload: {
  "iss": "holylabs",
  "aud": "<inbox.id>",
  "sub": "<host_user_id>",            // stable id from host's user system
  "name": "Roman Dr.",                // optional
  "email": "roman@example.com",       // optional
  "avatar_url": "https://…",          // optional
  "allowed_kinds": ["support", "order"],
  "external_refs": {                  // optional, kind → allowlist
    "order": ["ord_123"]
  },
  "iat": 1747861200,
  "exp": 1747864800                   // iat + ≤ 3600
}
```

- **`alg`**: HS256 only. The `kid` header tells the verifier
  which `inboxes` row to load the signing key from. Any other
  `alg` → 401.
- **`exp` cap**: 60 minutes from `iat`. The mint endpoint clamps
  any caller-requested `ttl_seconds` to `[300, 3600]`.
- **`aud`** must equal `inboxes.id` of the inbox the
  `sk_live_…` belongs to. Cross-inbox tokens are forbidden.
- **`sub`** is the host's user id — opaque string up to 256
  chars. Used as `external_ref` for support-kind conversations,
  and as the participant identity everywhere else.

### 2.4 `inboxes.widget_signing_secret` *(round 5)*

The HMAC key the mint endpoint signs JWTs with.

- `widget_signing_secret` — `bytea` not null. 32 random bytes.
- `widget_signing_secret_previous` — `bytea` nullable. For
  signature rotation: tokens minted before rotation continue to
  verify until the previous key is cleared.

Rotation policy is **separate** from `sk_live_` rotation. Rotating
the JWT-signing key invalidates issued tokens silently after the
previous key is cleared; rotating `sk_live_` only stops minting,
not verifying.

### 2.5 Routing identity

`/api/embed/customer/*` requires **both**:

- `Authorization: Bearer <widget JWT>`
- `x-holylabs-pk: <pk_live_…>`

The pk identifies the inbox; the JWT identifies the user. The
verifier confirms `JWT.aud === pk → inbox.id`. Both checks must
pass; a JWT with no pk header, or a pk with no JWT, is 401.

## 3. The schema (final shape — migration 0025 creates exactly this)

### 3.1 `inboxes` (extended)

```sql
alter table inboxes
  add column if not exists server_secret_hash            text,
  add column if not exists server_secret_previous_hash   text,
  add column if not exists server_secret_rotated_at      timestamptz,
  add column if not exists widget_signing_secret         bytea,
  add column if not exists widget_signing_secret_previous bytea,
  add column if not exists auth_mode                     text not null
                                                          default 'authenticated';

-- Lock the column to 'authenticated' this round. Round 6 will
-- extend the check to include 'anonymous'.
alter table inboxes
  add constraint inboxes_auth_mode_check
  check (auth_mode in ('authenticated'));

-- Distinct-secret check mirrors 0024_inbox_webhook_signing.sql.
alter table inboxes
  add constraint inbox_server_secret_distinct
  check (
    server_secret_previous_hash is null
    or server_secret_hash is null
    or server_secret_hash <> server_secret_previous_hash
  );
```

Backfill: every existing inbox gets a fresh `widget_signing_secret`
in the migration body (`update … set widget_signing_secret =
gen_random_bytes(32) where widget_signing_secret is null`). No
backfill for `server_secret_hash` — owners mint their `sk_live_`
from the dashboard.

## 4. JWT verification contract

Every `/api/embed/customer/*` handler validates as follows. The
implementation lives in `src/lib/widget-token.ts` (prompt 1
creates it).

```ts
type WidgetClaims = {
  iss: 'holylabs';
  aud: string;            // inbox.id
  sub: string;
  name?: string;
  email?: string;
  avatar_url?: string;
  allowed_kinds: ('support'|'order'|'direct')[];
  external_refs?: Record<string, string[]>;
  iat: number;
  exp: number;
};

export async function verifyWidgetToken(
  rawJwt: string,
  publishableKey: string,
): Promise<{ claims: WidgetClaims; inboxId: string; businessId: string }>;
```

Failure modes (every one returns 401 with `error: "invalid token"` —
no leaky detail to the browser):

- malformed JWT
- `alg` ≠ `HS256`
- signature fails against current key AND previous key (within
  grace window)
- `iss` ≠ `holylabs`
- `aud` ≠ pk's inbox id
- `exp` past
- `iat` in the future (skew > 60s)
- inbox not found, archived, or business suspended

Allowed-kind enforcement is per-handler: a create-conversation
handler asserts `kind ∈ claims.allowed_kinds` before insert.

External-ref enforcement is per-handler: when `claims.external_refs[kind]`
exists, the requested ref must be in the array; when it doesn't,
the implicit ref for `kind = 'support'` is `claims.sub`.

## 6. postMessage handshake

The host ↔ iframe bridge needs to be hardened (brief §8). The
narrow command set for round 5:

```
host → widget:
  { v: 1, type: 'init',  nonce: <random>, hostOrigin: <expected origin> }
  { v: 1, type: 'open',  kind?, externalRef?, startOptionId? }
  { v: 1, type: 'close' }
  { v: 1, type: 'sign-out' }                              // host user signed out; destroy session

widget → host:
  { v: 1, type: 'ready',  nonce: <echoed> }
  { v: 1, type: 'open',   open: boolean }
  { v: 1, type: 'unread', count: number }
```

Rules:

- The first message sent or received MUST be `init` / `ready`
  with a 16-byte random `nonce`. Subsequent messages must echo
  the same nonce in the same field on every message until the
  iframe is destroyed. (Prompt 2 wires the nonce into the
  bridge.)
- Every `message` listener filters by exact `event.origin ===
  hostOrigin` after the handshake. Before the handshake, the
  widget derives `hostOrigin` from `document.referrer`; if
  `referrer` is empty (rare; usually means `noreferrer` on the
  iframe), the widget refuses to render and posts an error to
  the parent.
- Unknown message `type` values are silently ignored. The widget
  never executes commands embedded in postMessage payloads
  except for the explicit set above.
- `sign-out` immediately tears down: closes the panel, clears
  any in-memory token, unsubscribes from realtime, and waits for
  a fresh `init` before doing anything else.

## 8. RLS recap

Every table added or extended in round 5 follows the same shape:

- `service_role` bypass (used by server actions and customer API).
- Browser anon role has **no** direct table access. Reads happen
  through `/api/embed/customer/*`; writes happen through dashboard
  server actions.
- `select` policy for the owner of the parent business, for
  dashboard preview purposes only.

No table in this round needs a write policy for the anon role —
the widget never queries Supabase directly from the browser.
