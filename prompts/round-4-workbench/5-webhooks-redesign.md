# 5 — Webhooks & integrations redesign

Read `AGENTS.md` and `0-shared.md` before starting. This prompt
is **independent of prompts 1–4** and can run in parallel after
`0-shared.md` is locked. Only the small "wire the
`conversation_assigned` event filter" step depends on prompt 4

## Goal

Rebuild `/dashboard/webhooks` into a modern integration surface:

1. **Dual-secret HMAC signing** (Stripe-style) — every outbound
   webhook carries an `X-Chatkit-Signature` header. Rotation
   keeps both secrets valid until the founder cuts over.
2. **Per-event opt-in** — tenants subscribe only to the events
   they want.
3. **Test-fire button** — sends a sample payload of any event
   to the configured URL and shows the response inline.
4. **Recent deliveries table** — last 50, with payload drawer.
5. **Per-business allowlist editor** — the schema landed in
   migration 0020 but the UI doesn't exist yet. Build it here
   on the business settings page (not the webhooks page).
6. **Embed integration recipes** — extend the existing
   `EmbedSnippets.tsx` to cover the FAB widget.

## Step 1 — Migration `0024_inbox_webhook_signing.sql`

alter table inboxes
  add column if not exists webhook_secret              text,
  add column if not exists webhook_secret_previous     text,
  add column if not exists webhook_secret_rotated_at   timestamptz,
  add column if not exists webhook_events              text[] not null default

    or webhook_secret is null
    or webhook_secret <> webhook_secret_previous

## Step 2 — Update `tenant-webhook.ts` to sign + filter

`src/lib/tenant-webhook.ts` currently fires unsigned and
unconditionally. Changes:

### 2a — Signature

For each fire, compute:

```
timestamp = floor(Date.now() / 1000)
signed_payload = `${timestamp}.${json_body}`
v1 = hmac_sha256(webhook_secret, signed_payload).toString('hex')
```

If `webhook_secret_previous` is set AND `webhook_secret_rotated_at`
is within the last 24 hours, also compute:

```
v1_prev = hmac_sha256(webhook_secret_previous, signed_payload).toString('hex')
```

Set header:

```
X-Chatkit-Signature: t=<timestamp>,v1=<v1>[,v1=<v1_prev>]
```

Multiple `v1=` entries are accepted by Stripe-style verifiers
— tenants check both during rotation.

Document the algorithm at the top of `tenant-webhook.ts` in a
brief comment block (4–6 lines max). The docs page change
follows in step 5.

### 2b — Event filter

Before firing, check the inbox's `webhook_events` array:

```ts
if (!inbox.webhook_events?.includes(payload.event)) {
  return; // tenant has unsubscribed from this event
}
```

Default array (set in migration) includes all four current
events. Tenants opting out is a privileged action; defaults are
opt-in for backward compat.

### 2c — All event payloads carry a stable shape

Audit the existing payload types in `tenant-webhook.ts`
(`MessageReceivedPayload`, etc.). Ensure each has:
- `event: string` — the discriminator.
- `tenant_id: string`
- `inbox_id: string`
- `occurred_at: string` — ISO timestamp (server-side `now()`).

Add the missing field `occurred_at` to any payload that lacks
it. Document the change in the file's header comment.

### 2d — Webhook delivery logging

`webhook_deliveries` already exists (migration 0012). For each
attempt, log:
- `attempted_at`
- HTTP status code (or 0 if network error)
- response body, first 4 KB
- `event_kind` (new column? Check if `webhook_deliveries`
  already records the event name — if not, add it in 0024):

```sql
alter table webhook_deliveries
  add column if not exists event_kind text,
  add column if not exists payload jsonb;
```

Update `tenant-webhook.ts` to populate these.

## Step 5 — Documentation snippet

Update the docs panel that explains how to verify the
signature. Suggested copy (adapt to the dashboard's voice):

> **Verifying signatures**
>
> Each request carries `X-Chatkit-Signature: t=<unix
> seconds>,v1=<hex>[,v1=<hex>]`. To verify:
>
> 1. Split header into `t` and one-or-more `v1` values.
> 2. Reject if `|now - t| > 5 minutes`.
> 3. Compute `expected = HMAC_SHA256(secret, "{t}.{body}")` in
>    hex. The body is the raw request bytes, not re-encoded.
> 4. Compare each `v1` against `expected` using a
>    constant-time comparator.
> 5. Accept if any matches. During a rotation window both the
>    new secret and the previous secret will produce valid
>    signatures.

Include a copy-paste Node.js snippet. Keep it ≤ 20 lines.
